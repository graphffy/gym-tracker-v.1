# GTracker frontend

SPA-клиент для существующего Spring Boot backend Gym Tracker.

## API

По умолчанию клиент ходит в `/api/v1`. В режиме разработки Vite проксирует `/api` на `http://localhost:8080`, поэтому CORS-настройки backend не нужны.

Если backend запущен на другом адресе, создайте `frontend/.env`:

```bash
VITE_API_BASE_URL=http://localhost:8080/api/v1
```

## Запуск

Из корня проекта:

```bash
.\mvnw.cmd spring-boot:run
```

В другом терминале:

```bash
cd frontend
npm install
npm run dev
```

Откройте `http://localhost:5173`.

## Сборка

```bash
cd frontend
npm run build
```

## Что важно по backend-контракту

- Основные CRUD endpoint-ы используются как есть: `/api/v1/categories`, `/api/v1/exercises`, `/api/v1/users`, `/api/v1/workouts`, `/api/v1/sets`.
- Backend возвращает списки для категорий, упражнений, пользователей и тренировок без серверной пагинации, поэтому на этих страницах пагинация и фильтры сделаны на клиенте.
- У `/api/v1/sets/search/jpql` есть серверная пагинация по имени пользователя и упражнению, но для единого UX страница подходов использует общий список и клиентские фильтры по тренировке, упражнению и человеку. Лучше добавить серверную пагинацию и фильтры для всех списков, когда данных станет много.
- ID нигде не отображаются пользователю. Они используются только внутри кода для роутинга, выбора связанных сущностей и API-запросов.
